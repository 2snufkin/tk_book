Liquibase Developer Reference Guide

Introduction​

Liquibase is an open-source database schema change management solution that enables you to track, version, and deploy database changes reliably across different environments. This guide explains core concepts, configuration, and practical usage to help developers new to Liquibase integrate it into their workflow.

Core Concepts​

How Liquibase Works​

  1. Change Tracking: Liquibase uses XML, YAML, JSON, or SQL files to define database schema changes.
  2. Execution Flow:
    • Liquibase connects to your database
    • It acquires a lock (using DATABASECHANGELOGLOCK table)
    • It checks which changes have been applied (using DATABASECHANGELOG table)
    • It runs any pending changes in order
    • It records the execution in the tracking table
    • It releases the lock

Key Components​

1. Changelog Files​

Changelog files are the heart of Liquibase. They contain a sequence of changesets that define database modifications:

<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
                   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
                   http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-3.10.xsd">

    <changeSet id="create-users-table" author="developer">
        <createTable tableName="USERS">
            <column name="id" type="INT">
                <constraints primaryKey="true" nullable="false"/>
            </column>
            <column name="username" type="VARCHAR(50)">
                <constraints unique="true" nullable="false"/>
            </column>
            <column name="created_date" type="DATETIME" defaultValueComputed="CURRENT_TIMESTAMP"/>
        </createTable>
    </changeSet>
</databaseChangeLog>

2. Changesets​

A changeset is an atomic unit of change that contains:

  • id: Unique identifier for the change
  • author: Developer who created the change
  • Changes: The actual database modifications (createTable, addColumn, etc.)
  • Optional elements: Preconditions, rollback instructions, contexts, etc.

3. Tracking Tables​

Liquibase creates and maintains two special tables:

  • DATABASECHANGELOG: Records all applied changesets (what, when, by whom)
  • DATABASECHANGELOGLOCK: Prevents multiple instances from running simultaneously

Configuration​

Properties File Setup​

The liquibase.properties file contains database connection information and execution parameters:

# Database connection
driver=com.mysql.cj.jdbc.Driver
url=jdbc:mysql://localhost:3306/your_database
username=your_username
password=your_password

# Changelog location
changeLogFile=db/changelog/db.changelog-master.xml

# Optional settings
logLevel=INFO
contexts=development

Command-line vs. Properties File​

While properties files provide a convenient way to store configuration, they should be used carefully in version control. Best practices include:

  1. Override via command line: Use command-line parameters to override sensitive properties
  2. Environment variables: Reference environment variables for credentials
  3. Multiple property files: Maintain separate files for different environments

Example command-line override:

liquibase --url="jdbc:mysql://localhost:3306/database" --username=dev --password=devpass update

Integration with Build Tools​

Maven Integration​

<plugin>
    <groupId>org.liquibase</groupId>
    <artifactId>liquibase-maven-plugin</artifactId>
    <version>4.9.1</version>
    <configuration>
        <propertyFile>src/main/resources/liquibase.properties</propertyFile>
    </configuration>
</plugin>

Spring Boot Integration​

@Bean
public SpringLiquibase liquibase(DataSource dataSource) {
    SpringLiquibase liquibase = new SpringLiquibase();
    liquibase.setDataSource(dataSource);
    liquibase.setChangeLog("classpath:db/changelog/db.changelog-master.xml");
    liquibase.setContexts("development,test");
    return liquibase;
}

Change Types and Examples​

Table Operations​

<!-- Create a new table -->
<changeSet id="create-products-table" author="developer">
    <createTable tableName="PRODUCTS">
        <column name="id" type="INT" autoIncrement="true">
            <constraints primaryKey="true" nullable="false"/>
        </column>
        <column name="name" type="VARCHAR(255)"/>
        <column name="price" type="DECIMAL(10,2)"/>
    </createTable>
</changeSet>

<!-- Add a column -->
<changeSet id="add-description-column" author="developer">
    <addColumn tableName="PRODUCTS">
        <column name="description" type="TEXT"/>
    </addColumn>
</changeSet>

<!-- Drop a table -->
<changeSet id="drop-old-table" author="developer">
    <dropTable tableName="LEGACY_TABLE"/>
</changeSet>

Data Operations​

<!-- Insert data -->
<changeSet id="insert-test-data" author="developer">
    <insert tableName="USERS">
        <column name="id" value="1"/>
        <column name="username" value="admin"/>
    </insert>
</changeSet>

<!-- Load data from file -->
<changeSet id="load-product-data" author="developer">
    <loadData tableName="PRODUCTS" file="data/products.csv">
        <column name="id" type="NUMERIC"/>
        <column name="name" type="STRING"/>
        <column name="price" type="NUMERIC"/>
    </loadData>
</changeSet>

Using SQL Directly​

<!-- Custom SQL -->
<changeSet id="complex-query" author="developer">
    <sql>
        CREATE VIEW active_users AS
        SELECT id, username FROM users WHERE status = 'active';
    </sql>
    <rollback>DROP VIEW active_users;</rollback>
</changeSet>

Advanced Features​

Preconditions​

Preconditions validate that certain conditions are met before applying a changeset:

<changeSet id="add-index" author="developer">
    <preConditions onFail="MARK_RAN">
        <tableExists tableName="USERS"/>
        <columnExists tableName="USERS" columnName="email"/>
    </preConditions>
    <createIndex tableName="USERS" indexName="idx_user_email">
        <column name="email"/>
    </createIndex>
</changeSet>

Contexts​

Contexts allow you to control when changesets are executed:

<changeSet id="add-test-data" author="developer" context="test">
    <insert tableName="USERS">
        <column name="id" value="999"/>
        <column name="username" value="test_user"/>
    </insert>
</changeSet>

Rollback Support​

Explicit rollback instructions help Liquibase revert changes safely:

<changeSet id="add-status-column" author="developer">
    <addColumn tableName="USERS">
        <column name="status" type="VARCHAR(20)" defaultValue="active"/>
    </addColumn>
    <rollback>
        <dropColumn tableName="USERS" columnName="status"/>
    </rollback>
</changeSet>

Troubleshooting​

Common Issues and Solutions​

Checksum Mismatch​

Problem: When a changeset is modified after being applied, Liquibase detects a checksum mismatch.

Solution: Clear checksums for a specific file or all files:

# Clear all checksums
liquibase clearCheckSums

# Clear checksums for a specific file
liquibase --changeLogFile=db/changelog/specific-file.xml clearCheckSums

Database Lock Issues​

Problem: If Liquibase terminates unexpectedly, it might leave the database locked.

Solution: Manually release the lock:

-- Check lock status
SELECT * FROM DATABASECHANGELOGLOCK WHERE ID = 1;

-- Release the lock
UPDATE DATABASECHANGELOGLOCK SET LOCKED = 0, LOCKGRANTED = NULL, LOCKEDBY = NULL WHERE ID = 1;

Path Resolution Problems​

Problem: SQL files referenced in changesets with relative paths cannot be found.

Solution: Use the --searchPath parameter to specify additional directories:

liquibase --searchPath="path/to/sql/files,path/to/resources" update

Best Practices​

  1. One Change Per Changeset: Keep changesets small and focused on a single logical change
  2. Use IDs Consistently: Create a naming convention for changeset IDs
  3. Include Rollback Instructions: Provide explicit rollback steps where possible
  4. Test Before Production: Run changes in development and testing environments first
  5. Use Contexts Appropriately: Separate environment-specific changes
  6. Maintain a Master Changelog: Use a master file that includes other changelog files
  7. Version Control: Keep changelog files in the same repository as application code
  8. CI/CD Integration: Automate database changes as part of your deployment pipeline

Conclusion​

Liquibase provides a robust framework for managing database schema changes. By tracking changes in version control and automating their application, it eliminates many of the risks associated with database migrations. For developers new to the tool, focus on understanding changesets, proper configuration, and testing your changes thoroughly before applying them to production.
